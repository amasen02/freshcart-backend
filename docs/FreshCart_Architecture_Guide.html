The system in one picture

Top-to-bottom view of the platform — edge, gateway, services, persistence, eventing, observability.

The five tenets

Every decision in the repo is checked against these. The interview question "why?" always lands here.

  1. One demonstrable pattern per service. Each bounded context is implemented in a different architectural style so the repo doubles as a teaching artifact.
  2. Database per service. No cross-service joins. Cross-service reads use API aggregation at the gateway or a materialised read model.
  3. Sync hops are at most one deep. Beyond that, services communicate via events with the transactional outbox guarantee.
  4. Cookie-first for browsers, JWT for everything else. Browser sessions never see a JWT. Mobile and service-to-service callers never see a session cookie.
  5. Observability is not an add-on. Every service emits OpenTelemetry traces + metrics + logs from day one. If you cannot see it, you cannot operate it.

Evolution ladder — where this system sits

The repo deliberately demonstrates the climb from monolith to event-driven microservices, with documented exit criteria for each stage. The current implementation lives at stage 11 + 12: event-driven microservices with the transactional outbox and HybridCache, alongside one explicit modular monolith for the back-office.

  1. Monolith, one app, in-memory — MVP, <100 req/s. Documented under docs/evolution/.
  2. Monolith + persistent DB — first real users.
  3. Monolith + UI — real product, single deploy unit.
  4. Modular Monolith — 4–15 devs, clear bounded contexts. → AdminBackoffice.
  5. + Distributed cache — read-heavy.
  6. Microservices (first split) — multiple teams, scaling pressure.
  7. + Sync calls (HTTP / gRPC) — cross-context reads.
  8. + API Gateway (YARP) — public clients, auth, rate limit.
  9. + Event-driven (RabbitMQ + MassTransit) — eventual consistency.
  10. + Hybrid Cache (L1 + L2) — hot reads dominate.
  11. + Transactional Outbox — exactly-once event delivery — current.
  12. + Serverless slices — variable load — optional, future.
  13. + Sharding & multi-region — multi-region active-active when single-region IOPS hits the wall.

The domain — online supermarket

Domain decomposed from an event-storm of the customer journey: browse → add to basket → check out → pay → reserve stock → fulfil → deliver → notify → review → ask for support. Each stable noun and verb became either an aggregate, a domain event, or — when the cohesion was high and the coupling was low — a separate bounded context.

Bounded contexts

Twelve bounded contexts. Each owns its own data; cross-context state propagates as events. The colour code follows through to the per-service flow diagrams in the Animated Flows tab.

Identity A07

Users, roles, sign-up, sign-in, password reset, MFA, refresh, sessions. The security boundary every other service depends on.

Catalog Reads

Products, categories, brands, attributes, search, images. Read-heavy with separate scaling profile.

Pricing gRPC

Discounts, coupons, loyalty multipliers, time-window promos. Stateless hot path.

Basket Cache

Per-user shopping cart with TTL. Hot reads via HybridCache, writes go through transactional outbox.

Ordering DDD · Saga

Order lifecycle and checkout saga orchestration. Rich domain with multiple invariants.

Inventory Strong

Stock levels, reservations, replenishment. Strong consistency, transactional.

Payment PCI

Payment intents, gateway integration, refunds, audit trail. Event-sourced for compliance.

Delivery Geo

Slots, routes, driver assignment, ETA, geo-fencing. Geo-heavy.

Notification SignalR

Email + SMS + push + in-app real-time. Fan-out from event bus.

CustomerSupport WebSocket

Live agent ↔ customer chat. Persistent connection workload with sticky sessions.

Reviews Mongo

Product reviews, ratings, moderation. Doc-shaped data, eventual consistency.

Reporting OLAP

KPI dashboards, RDLC reports, analytics warehouse. Separate engine — OLAP isolation.

The 12 microservices

Each service is described with: domain, architectural style, persistence, communication style, functional and non-functional requirements, key technologies and patterns, plus a code-link.

Clean Architecture Identity Service

.NET 10ASP.NET IdentityEF CoreArgon2idJWTCookiesSQL Server

The security boundary every other service depends on. First service implemented end-to-end because every other service consumes its tokens.

Domain (Functional)
  • Customer self-service sign-up with email confirmation.
  • Cookie-first sign-in for browser SPAs; JWT issue for mobile and service-to-service.
  • Refresh-token rotation with reuse detection — token theft triggers family-wide revocation.
  • TOTP-based multi-factor authentication enrolment + verification.
  • Lockout after five failed sign-ins for fifteen minutes.
  • Password reset by signed time-limited token sent to verified email.
  • Role-based access: Customer, SupportAgent, Administrator.
  • Append-only audit log for every authentication event.
Non-functional
  • Sign-in p95 < 300 ms at 200 req/s steady-state.
  • Argon2id password hashing (64 MiB memory, 3 iterations, parallelism 4).
  • OWASP A07 — lockout, MFA, HttpOnly+Secure+SameSite=Strict cookies, anti-forgery.
  • Refresh-token reuse detection revokes the entire token family for that user.
  • Horizontal scaling via Redis-backed Data Protection key ring (cookie validation works behind a load balancer).
Tech & patterns
  • Clean Architecture — Domain / Application / Infrastructure / Api projects.
  • CQRS + MediatR with ValidationBehavior and LoggingBehavior.
  • FluentValidation per command; CustomExceptionHandler → ProblemDetails.
  • EF Core with retry-on-failure; migrations in the identity schema.
  • Carter modules for endpoint routing.
  • Cookie scheme + JWT bearer scheme co-exist; policy RequireAuthenticatedUser works against either.
Persistence

Azure SQL Database. Tables: identity.AspNetUsers, identity.AspNetRoles, identity.RefreshTokens, identity.AuditEvents.

Vertical Slice Catalog Service

.NET 10CarterMediatRMartenPostgreSQLRedis HybridCache

Product browsing, search and admin CRUD. The CRUD-heavy nature of the workload makes Vertical Slice a better fit than Clean — each feature is a self-contained folder.

Functional
  • Create / update / delete / get / search products with categories, brands, attributes.
  • Image upload to Azure Blob Storage via signed URLs (Browser → Blob direct).
  • Price change publishes ProductPriceChangedIntegrationEvent.
  • Full-text search via Postgres trigram index, accelerated by Redis cache.
Non-functional
  • Read p95 < 80 ms; cache hit ratio > 90% in steady state.
  • Idempotent Create accepts Idempotency-Key header.
  • Horizontally scalable with no sticky-session requirement.
Tech & patterns
  • Vertical Slice — Features/{Aggregate}/{UseCase}/{Command|Query|Handler|Endpoint|Validator}.cs.
  • Marten over Postgres for document-store ergonomics on a relational engine.
  • HybridCache for L1 (in-process) + L2 (Redis).
  • Mapster for DTO ↔ command projection.

gRPC Pricing Service

.NET 10gRPCProtobufSQLite (in-container)

Stateless hot-path price computation: discounts, coupons, loyalty multipliers, time-window promotions. gRPC chosen for strongly-typed contract and lower per-call cost.

Functional
  • GetPriceQuote — applies all active rules to a basket snapshot and returns the priced lines.
  • ValidateCoupon — single-call check used at basket time.
  • StreamLivePrices — server-streaming RPC for the admin pricing console.
Non-functional
  • p99 < 25 ms — stateless, in-memory rule cache, SQLite ships in the container.
  • Backward-compatible proto evolution only — additive fields, never renumber.
  • Available on both http2 (gRPC) and http1 (gRPC-Web) for browser fallback.

Vertical Slice + Outbox + Decorator Basket Service

.NET 10PostgreSQLRedis HybridCacheMassTransitOutbox

The shopping cart. Hot user state — every page view touches it. Cross-service writes (checkout, price update from Catalog) go through the transactional outbox.

Functional
  • Add / update / remove item with idempotent client-supplied basket id.
  • Sync with Pricing on every change for line totals.
  • Consume ProductPriceChangedIntegrationEvent from Catalog to refresh line prices.
  • Checkout publishes BasketCheckoutIntegrationEvent via outbox; Ordering picks it up.
Non-functional
  • Read p95 < 30 ms (HybridCache).
  • Exactly-once event delivery on checkout — transactional outbox + idempotent consumer in Ordering.
  • Basket TTL 30 days; abandoned-cart event fires at 24h.
Patterns
  • Vertical Slice per use-case.
  • Decorator — CachedBasketRepository wraps BasketRepository; registered via Scrutor's services.Decorate<...>().
  • Outbox — business write + OutboxMessage insert in one EF transaction; OutboxPublisher : BackgroundService drains it.
  • HybridCache — L1 in-proc + L2 Redis under a single API.

Clean / Onion + DDD + Saga Ordering Service

.NET 10EF Core (write)Dapper (read)SQL ServerMassTransit SagaOutbox

The richest domain in the platform. Order is a true DDD aggregate root with multiple invariants (items must sum to total; total must equal payment intent; ship and bill addresses captured as value objects). Checkout is an orchestrated saga.

Functional
  • Consume BasketCheckoutIntegrationEvent → start checkout saga.
  • Saga steps: ReserveStock (Inventory gRPC) → ChargePayment (Payment REST) → ReserveDeliverySlot (Delivery REST) → ConfirmOrder.
  • Each step has a compensating action on failure.
  • Domain events: OrderCreated, OrderPaid, OrderShipped, OrderCancelled → published via outbox.
  • Queries: GetOrderById, GetOrdersByCustomer, GetOrdersByName — Dapper-backed.
Non-functional
  • Checkout p95 < 1.5 s end-to-end (gateway → saga → confirmation).
  • Saga state durable in SQL Server; survives pod restart.
  • Exactly-once domain-event publication via outbox + dispatch interceptor on SaveChanges.
Patterns
  • DDD Aggregate (Order), Value Objects (OrderId, Address, Payment, OrderName), Domain Events.
  • Hybrid data access — EF Core for write side, Dapper for read side (CV claim demonstrated here).
  • Saga (orchestrated) — MassTransit state machine.
  • EF Core SaveChangesInterceptor — AuditableEntityInterceptor + DispatchDomainEventsInterceptor.
  • Outbox for cross-service integration events.

Layered (3-tier) + Dapper Inventory Service

.NET 10DapperSQL ServergRPC + REST

Stock management — transactional, perf-critical, schema-stable. A classic Layered architecture (Controllers → Services → Repositories → DataAccess) fits perfectly without over-engineering.

Functional
  • gRPC ReserveStock / ReleaseStock consumed by Ordering saga.
  • REST endpoints for admin replenishment, stock-take audit.
  • Low-stock event published when threshold crossed.
Non-functional
  • Reservation p99 < 50 ms.
  • Strong consistency — every reservation is a single SQL transaction with row-level lock.
  • Dapper used for hot reads (stock-level dashboards) — read p99 < 30 ms.

Clean + Event Sourcing Payment Service

.NET 10MongoDB (event store)SQL Server (read projection)External gateway

PCI-isolated. Event-sourced because every change must be auditable for compliance, and the read model can be rebuilt from the event log at any time.

Functional
  • Create payment intent → external gateway → capture → settle (or refund).
  • Every state change is an event: PaymentIntentCreated, PaymentAuthorised, PaymentCaptured, PaymentRefunded.
  • Read projection consumes events and writes flat tables for the rest of the platform to query.
Non-functional
  • No PII other than tokenised card identifiers — never stores PAN.
  • Mutual TLS to the external gateway.
  • Append-only event store; the read projection is rebuildable.
  • Idempotency-Key header is mandatory on every mutating endpoint.

Hexagonal (Ports & Adapters) Delivery Service

.NET 10MongoDB (2dsphere)Google Maps / Mapbox adapter

Delivery slots, routes, ETA, geo-fencing. Multiple external adapters (Maps provider, driver mobile app, customer SPA) — Hexagonal isolates the domain from each adapter so they can be swapped independently.

Functional
  • Reserve a delivery slot for an order (consumed by Ordering saga).
  • Build an optimal route per driver per day (BackgroundService).
  • Driver app polls assigned stops and reports completions.
  • Customer SPA shows live ETA via SignalR push from Notification service.

Pipes-and-filters + SignalR Notification Service

.NET 10SignalRMassTransit consumerCosmos DBRedis backplane

Single sink for every customer-facing notification. Consumes integration events, decides channel per recipient (in-app via SignalR, email, SMS, push), and persists a delivery record in Cosmos.

Functional
  • Consume every Order*, Payment*, Delivery* integration event.
  • Resolve recipient preferences (in-app + email by default).
  • Push to the connected browser via the NotificationHub SignalR hub.
  • Persist the notification record for later in-app retrieval.
Non-functional
  • SignalR backplane = Redis — works across N pods on AKS.
  • p95 from event published → browser receives push < 1 s.
  • Cosmos DB chosen for globally distributed low-latency writes.

Stateful WebSocket CustomerSupport Service

.NET 10SignalR over WebSocketMongoDB transcriptsRedis presenceSticky session

Live chat between customers and support agents. Persistent bi-directional WebSocket; sticky sessions configured on the AKS ingress so each customer remains pinned to a single pod for the duration of the conversation.

Functional
  • Customer opens chat → routed to first available agent in their region.
  • Agent console aggregates active chats and assigns priorities.
  • Full transcript persisted to MongoDB; replays available for compliance.
  • Typing indicators + read receipts.

Vertical Slice + MongoDB Reviews Service

.NET 10MongoDBMediatR

Product reviews and ratings. Doc-shaped data with optional media attachments. Schemaless suits the use case — review templates evolve frequently.

Functional
  • Authenticated customers submit one review per product after purchase.
  • Moderation pipeline: pending → published / rejected.
  • Aggregation pipeline computes star averages on the fly.

CQRS read model + RDLC Reporting Service

.NET 10MySQL warehouseRDLCDapper

OLAP read model isolated from OLTP. Consumes integration events from every service and projects denormalised dashboards.

Functional
  • Daily sales, top products, slow-moving stock, customer LTV.
  • RDLC report generation for PDF / Excel exports.
  • Pre-aggregated dashboards refreshed on event-arrival.

Edge + BFF YARP API Gateway

.NET 10YARPRate limiterCookie validation

Single ingress for the SPAs. Routes by path prefix (/catalog-service/*, /basket-service/*, …), validates the session cookie or bearer token, applies a per-route fixed-window rate limiter, and provides BFF-aggregator endpoints (/bff/checkout-summary) that compose multiple downstream calls.

Modular Monolith AdminBackoffice

.NET 10IModule patternSingle process

Demonstrates the step before microservices. Three modules (CatalogAdmin, InventoryAdmin, Reports) live in one process with Add{X}Module() + Use{X}Module() extension methods. Each module owns its schema; cross-module calls go through public module APIs, never direct DbContext access.

Patterns & principles in use

Every pattern below is anchored to a concrete location in the codebase. Picked at the right granularity — not every CRUD endpoint needs DDD, not every script needs Clean Architecture.

Architectural patterns

Clean Architecture

Identity, Ordering, Payment. Domain at the centre; Application defines ports; Infrastructure implements them; Api is the composition root.

Onion Architecture

Effectively the same shape as Clean; used as the explicit name for Ordering to match the candidate's CV vocabulary.

Vertical Slice

Catalog, Basket, Reviews. Features/{Aggregate}/{UseCase}/{Command,Query,Handler,Endpoint,Validator}.cs.

Layered (3-tier)

Inventory. Controllers → Services → Repositories → DataAccess. The right choice when the domain is stable and CRUD-heavy.

Hexagonal (Ports & Adapters)

Delivery. Multiple external adapters (Maps provider, driver app, customer notifications) isolate the domain from each.

Modular Monolith

AdminBackoffice. IModule per area; one process; can be extracted to microservices later with minimal change.

API Gateway / BFF

YARP. Single ingress, routes, rate limit, cookie validation, BFF aggregator endpoints.

Sidecar (via Aspire)

The Aspire AppHost models backing services as orchestrated sidecars in DEV; in PROD the same role is played by AKS namespaces + Helm releases.

Application-level patterns

CQRS

Every service. ICommand<T> / IQuery<T> + dedicated handlers via MediatR.

Mediator

MediatR + ISender at endpoints; never call handlers directly.

Pipeline Behavior

ValidationBehavior, LoggingBehavior wrap every request in a generic, zero-per-handler cross-cutting layer.

Domain-Driven Design

Ordering — Aggregate (Order), Value Objects (OrderId, Address, Payment), Domain Events.

Strongly-Typed IDs

OrderId.Of(guid) — compile-time prevention of mixing a CustomerId with an OrderId.

Repository + Unit of Work

Ordering, Inventory. DbContext is the UoW — no second abstraction.

Decorator

Basket — CachedBasketRepository wraps BasketRepository via Scrutor.

Specification

Inventory — composable read-side predicate objects passed into Dapper queries.

Result envelope

Used in the pure domain where a domain exception would be too coarse.

Integration & messaging patterns

Transactional Outbox

Basket, Ordering. Business write + OutboxMessage insert in one transaction; OutboxPublisher : BackgroundService drains it. At-least-once delivery → consumers must be idempotent.

Inbox (idempotent consumer)

Ordering, Payment — every consumer checks the inbox before applying state changes.

Saga (orchestrated)

Ordering — MassTransit MassTransitStateMachine<CheckoutState>. Each step has a compensating action.

Saga (choreographed)

Notification — pure event consumer, no central orchestrator needed.

Anti-Corruption Layer

Payment — the external gateway has a different model; an ACL keeps the gateway's terminology out of the domain.

Event Sourcing

Payment — Mongo event store. Read model rebuildable.

Materialised Read Model

Reporting — subscribes to every service's events and projects denormalised views.

Pub/Sub + Queue

RabbitMQ topics for pub/sub; queues per consumer for work distribution.

Resilience & performance patterns

Retry with exponential backoff

AddStandardResilienceHandler() on every typed HttpClient.

Circuit Breaker

Same bundle, fails fast after threshold; half-opens after cool-down.

Timeout

Per-call deadline; combined with retry to avoid runaway latency.

Bulkhead

Isolated connection pools per downstream service.

Fallback

Catalog returns last-known-good cache entry when Pricing is degraded.

HybridCache (L1 + L2)

.NET 9+. Sub-ms in-proc + cross-instance Redis under a single API.

Rate Limiting

YARP fixed-window limiter per route; per-client keyed limiter for write endpoints.

Idempotency Key

Required on every mutating endpoint that may be retried.

Authentication flow — cookie-first browser sign-in

Why HttpOnly + Secure + SameSite=Strict beats a JS-readable JWT

  • HttpOnly — JavaScript cannot read the cookie.
  • Secure — cookie sent only over HTTPS.
  • SameSite=Strict — cookie not sent on cross-site requests; defeats most CSRF without anti-forgery.
  • Anti-forgery double-submit — defence-in-depth: XSRF-TOKEN cookie + X-XSRF-TOKEN header must match.
  • Sliding expiration — 8 hours idle; full re-auth after 24 hours.
  • Server-side invalidation — sign-out updates security stamp; existing cookies are invalidated on next request.

Service-to-service authentication — JWT client credentials

  • HS256-signed JWT (signing key in Azure Key Vault; rotated quarterly).
  • Access token lifetime: 15 minutes. Refresh token lifetime: 14 days.
  • Refresh-token rotation on every refresh; reuse detection revokes the entire token family.
  • Plaintext token returned exactly once at issue/rotate; SHA-256 hash persisted.
  • Per-endpoint authorisation policies — Customer, SupportAgent, Administrator — apply to either scheme.

Coding conventions (excerpt)

Full version: docs/CONVENTIONS.md.

  • Names are full words. productCatalogRepository, never repo. pendingOutboxMessages, never pendingMsgs. No single-letter locals outside the tightest of loops.
  • Methods read like sentences. await refreshTokenService.RotateAsync(...) describes intent; ProcessAsync does not.
  • No filler comments. Comments explain the why; well-named identifiers explain the what.
  • Records for DTOs, commands, queries, value objects. Classes for entities, aggregates, services.
  • Every async method takes a CancellationToken and forwards it.
  • LINQ stays in IQueryable until the explicit materialisation call (ToListAsync, FirstOrDefaultAsync).
  • AsNoTracking() on every read path. Tracking is opt-in.
  • Constructor injection only. No service locator.
  • Conventional Commits — feat(catalog): add product search vertical slice.

Adding a new feature — the cookbook

  1. Identify the bounded context the feature belongs to. If it crosses two contexts, model the cross-context part as an integration event.
  2. Decide command or query. Command = write; Query = read.
  3. Create the feature folder under the service's Features/<Aggregate>/<UseCase>/ (Vertical Slice) or under Application/<Aggregate>/Commands|Queries/ (Clean Architecture).
  4. Write the record (Command / Query) and its handler. Add a FluentValidation validator.
  5. For Clean Architecture services, declare any new infrastructure ports as I… interfaces in Application/Common/Abstractions. Implement them in Infrastructure.
  6. Write the Carter endpoint that dispatches via ISender.
  7. Write the unit test for the handler. Write the integration test for the endpoint (Testcontainers).
  8. If the change emits an integration event, add the OutboxMessage insert in the same EF transaction.
  9. Update the relevant ADR or interview-tour card if the design decision is novel.

Invoice numbering — gap-free per (year, kind)

Tax jurisdictions require gap-free, monotonically-increasing invoice numbers within a year per document kind. FreshCart implements this with an invoice_number_sequences table and an UPDATE under InnoDB row lock:

// InvoiceRepository.AllocateNextNumberAsync
var sequenceRow = await dbContext.InvoiceNumberSequences
    .FirstOrDefaultAsync(s => s.Year == year && s.Kind == kind, cancellationToken);

sequenceRow ??= new InvoiceNumberSequence { Year = year, Kind = kind, LastSequence = 0 };
sequenceRow.LastSequence += 1;
await dbContext.SaveChangesAsync(cancellationToken);   // implicit row lock until commit

return InvoiceNumber.Allocate(kind, year, sequenceRow.LastSequence);

Format examples: INV-2026-000123, CR-2026-000018, PF-2026-000009. InvoiceNumber.TryParse in the domain layer round-trips them.

Signal-store pattern (used per feature)

@Injectable({ providedIn: 'root' })
export class BasketStore {
  private readonly basketService = inject(BasketService);

  readonly items = signal<BasketItem[]>([]);
  readonly itemCount = computed(() => this.items().length);
  readonly subtotal = computed(() =>
    this.items().reduce((total, item) => total + item.lineTotal, 0));
  readonly isEmpty = computed(() => this.itemCount() === 0);

  async addItem(productId: string, quantity: number): Promise<void> {
    const updatedBasket = await firstValueFrom(
      this.basketService.addItem(productId, quantity));
    this.items.set(updatedBasket.items);
  }
}

Anti-forgery on the SPA side

// core/http/csrf.interceptor.ts
export const csrfInterceptor: HttpInterceptorFn = (request, next) => {
  const isMutating = ['POST', 'PUT', 'PATCH', 'DELETE'].includes(request.method);
  if (!isMutating) return next(request);

  const xsrfToken = inject(CookieService).get('XSRF-TOKEN');
  if (!xsrfToken) return next(request);

  return next(request.clone({
    setHeaders: { 'X-XSRF-TOKEN': xsrfToken }
  }));
};
